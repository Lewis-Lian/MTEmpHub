# 前后端分离设计文档

## 背景

当前系统是基于 Flask 的单体应用，`routes/` 同时承担页面渲染与数据接口职责，`templates/` 和 `static/js/` 直接服务业务页面。用户希望将系统改造成完整的前后端分离架构，并明确选择：

- 采用一次性重构方案，而不是长期双轨迁移
- 前端技术栈使用 `React + Vite`
- 生产环境按分域部署

本设计文档用于定义目标架构、迁移边界、切换方式和验证标准，为后续实施计划提供唯一依据。

## 目标

将当前系统重构为：

- 一个只提供业务 API、鉴权、权限校验、文件下载和健康检查的 Flask 后端
- 一个独立的 `React + Vite` 前端应用，负责所有业务页面、前端路由和交互

重构完成后：

- 后端不再承担业务页面渲染
- 新前端覆盖当前主要业务页面和后台管理页面
- 生产环境以前端域名和后端域名分离的方式运行

## 不做的事

本次重构不包含以下范围：

- 不在本次拆分中将 Flask 替换为其他后端框架
- 不在本次拆分中重写核心统计和导入业务逻辑
- 不在本次拆分中引入移动端、小程序或第三方客户端
- 不在本次拆分中顺手把现有权限模型改造成全新 RBAC 体系
- 不在本次拆分中额外引入 JWT 体系

## 方案选择

评估过的方案包括：

1. 渐进迁移
2. 一次性重构
3. 后端先 API 化，再启动前端

最终选择方案 2：一次性重构。

选择原因：

- 目标架构边界最清晰，不会长期维护老模板页和新前端两套 UI
- 可以在一个明确的切换窗口内完成入口替换，减少长期混合态复杂度
- 适合当前希望直接完成前后端彻底分离的目标

对应代价：

- 切换前必须完成主要页面、权限、导出和联调验证
- 切换窗口内风险集中，需要更严格的接口契约和回滚预案

## 目标架构

### 后端职责

Flask 后端保留以下职责：

- 提供 `/api/*` 业务接口
- 提供登录态管理与会话校验
- 提供权限校验
- 提供文件导出与下载
- 提供 `/health` 健康检查

Flask 后端不再负责：

- 业务页面 HTML 渲染
- 前端页面跳转与模板布局
- 新增基于 `templates/` 的业务页面

### 前端职责

独立前端负责：

- 登录页
- 查询中心页面
- 后台管理页面
- 菜单与前端路由
- 页面内筛选、表格、弹窗、状态展示
- 统一 API 请求封装、错误提示和权限可见性处理

### 部署边界

生产环境按分域部署：

- 前端示例域名：`https://app.example.com`
- 后端示例域名：`https://api.example.com`

浏览器访问前端域名，前端再调用后端 API。后端不再对外承载业务页面入口。

## 后端设计

### 路由重组

现有 `routes/` 中同时存在页面路由与数据接口。重构后需要将路由职责收敛到 API 资源层，建议统一整理为 `/api/*` 路径空间。

目标状态示例：

- `/api/auth/*`
- `/api/employee/*`
- `/api/admin/*`
- `/api/account-sets/*`
- `/api/downloads/*`

原有 `render_template(...)` 对应的页面路由，在切换完成后应下线，或仅保留为明确的跳转占位，不再承载业务页面。

### 业务逻辑保留原则

现有 `models/` 和 `services/` 保持为核心业务层，不做与前后端分离无关的大规模重写。后端改造以“补 API 边界”和“统一返回契约”为主，避免重写成熟的统计、导入和导出逻辑。

### 接口契约

所有前端页面依赖的数据能力都必须先沉淀为稳定 API，包括但不限于：

- 登录与当前用户信息
- 查询中心各类筛选查询
- 账套列表与相关基础数据
- 管理人员统计查询
- 异常查询与打卡记录查询
- 部门工时查询
- Excel 下载任务或直接下载能力
- 后台账号、部门、班次、修正项维护

统一返回约定：

- 成功响应使用统一 JSON 结构或明确资源结构
- 失败响应返回统一错误格式
- 权限不足返回明确的鉴权或授权错误
- 参数错误返回可供前端直接展示或映射的错误信息

## 前端设计

### 工程结构

在仓库下新增独立前端工程：

- `frontend/`

建议结构：

- `frontend/src/pages/`
- `frontend/src/components/`
- `frontend/src/layouts/`
- `frontend/src/api/`
- `frontend/src/hooks/`
- `frontend/src/types/`
- `frontend/src/router/`
- `frontend/src/utils/`

### 页面覆盖范围

由于选择一次性重构，切换时前端必须完整覆盖至少以下业务入口：

- 登录页
- 查询中心首页
- 员工考勤汇总查询
- 异常查询
- 打卡记录查询
- 部门工时查询
- 管理人员考勤查询
- 管理人员加班或统计查询
- 管理人员年休查询
- 管理人员部门工时查询
- 下载入口
- 后台首页
- 后台账号管理
- 员工与部门管理
- 班次与修正项维护

如果某个旧页面在生产仍有使用价值，它必须在切换前被新前端承接，不能留在旧模板体系中作为漏网入口。

### 前端路由

前端使用 React 路由统一管理业务页面，例如：

- `/login`
- `/employee/...`
- `/admin/...`

页面级权限不再依赖后端渲染时隐藏菜单，而是由前端根据后端返回的用户权限信息决定菜单展示与页面入口可见性。真正的数据访问权限仍以后端校验为准。

## 鉴权与跨域设计

### 鉴权方案

本次继续使用基于 Cookie 的会话方案，不引入 JWT。

原因：

- 现有系统已经具备基于 Flask 的登录态与权限语义
- 后台管理场景更适合服务端会话控制
- 避免额外引入前端 Token 存储、刷新、失效同步等复杂度

### 分域场景要求

前后端分域后，后端需要：

- 仅允许受信任前端域名跨域访问
- 允许携带凭证
- 正确设置会话 Cookie 的安全属性
- 在 HTTPS 下运行

前端请求后端时必须显式携带凭证，以维持登录态。

### 权限模型

继续沿用现有用户、角色、页面权限、人员范围和部门范围控制模型。重构重点是把现有“页面访问权限”映射为：

- 前端菜单和页面入口控制
- 后端接口访问控制

前端只负责体验层的可见性控制，真正的权限裁决必须保留在后端。

## 迁移与切换设计

### 迁移原则

虽然是一次性重构，但内部实施仍应分阶段推进：

1. 梳理现有页面与数据能力映射
2. 补齐并稳定后端 API 契约
3. 完成独立前端页面实现
4. 在预发环境完成完整联调
5. 在正式环境切换窗口内替换入口

这里的“一次性”指正式入口切换一次完成，而不是开发过程无阶段。

### 切换策略

建议采用“短窗口切换”：

- 切换前，后端 API 已具备生产可用性
- 切换前，前端已在预发环境完成完整验证
- 切换时，将正式访问入口切到新前端域名
- 切换后，旧模板页不再作为正式业务入口使用

### 回滚边界

回滚能力应尽量停留在入口层，而不是数据层。推荐做法：

- 保留旧页面体系在切换窗口内的可回退能力
- 不以回滚数据库结构为首选方案
- 如果切换后出现重大问题，优先把入口切回旧前端访问路径

这要求切换前不要做与新前端强绑定、且无法兼容旧逻辑的数据层破坏性改动。

## 测试与验收

### 后端验证

后端必须补足接口级验证，至少覆盖：

- 登录与登录态校验
- 关键查询接口的正常响应与参数错误响应
- 权限不足时的拒绝响应
- 导出和下载接口可用性
- 关键后台管理接口的增删改查

### 前端验证

前端至少覆盖以下关键业务链路：

- 登录成功与失败提示
- 菜单按权限显示
- 关键筛选项加载
- 查询结果表格正确展示
- 空状态、错误状态、加载状态展示
- 下载按钮触发正确请求

### 端到端冒烟

切换前必须执行至少一轮端到端冒烟，确认以下路径可用：

- 登录
- 进入主要查询页面
- 完成一次典型查询
- 进入后台页面
- 完成一次典型维护操作
- 触发一次下载

### 成功标准

本次重构完成的判定标准为：

- 生产环境业务入口由独立 React 前端承载
- Flask 不再对外提供业务页面渲染
- 主要业务页面全部可在新前端完成操作
- 前端通过 API 调用完成全部核心查询与后台管理
- 权限校验在新架构下仍然有效
- 分域部署下登录态稳定可用
- 核心下载能力在新架构下正常工作

## 主要改动边界

预计会触及的主要区域：

- 后端应用入口与配置
- `routes/` 中页面路由与接口路由的重组
- 部分认证和权限相关逻辑
- 导出与下载接口的响应方式
- 新增 `frontend/` 工程及其构建配置
- 开发联调与生产部署配置
- 测试体系补充

明确不建议在本次范围内同时做的事情：

- 大规模重命名现有业务模型
- 为追求“更优雅”而重写成熟服务层
- 顺手重构所有历史页面样式

## 实施前提

在进入实施计划前，需要接受以下前提：

- 一次性重构需要预发环境完整联调
- 切换窗口需要业务方接受短时间集中发布
- 关键页面清单必须先冻结，避免实施中途不断扩 scope
- 如果现有旧模板页仍存在隐性使用路径，需要在实施前排查清楚

## 结论

本次前后端分离采用“一次性重构 + React 前端独立工程 + Flask 纯 API 后端 + 分域部署 + Cookie 会话鉴权”的组合方案。

实施时应坚持三条原则：

- 后端只做 API，不再继续承载业务页面
- 业务逻辑保留在现有服务层，避免与拆分目标无关的重写
- 正式切换一次完成，但开发和验证必须分阶段执行

后续应基于本文档编写详细实施计划，逐项拆分 API、前端页面、联调、测试和切换任务。
