# React 前端恢复旧版页面 UI 设计

## 背景

当前仓库的 React 前端已经替换了旧版 Flask 模板页面的主要入口，但界面骨架与视觉风格已经明显偏离原系统：

- 登录页改成了新的卡片式欢迎页
- 内页改成了新的深色侧边栏加卡片概览布局
- 查询首页和业务页大量使用新的大圆角卡片、渐变和营销化排版

用户本次要求是：**把前端页面 UI 做成原来的样子**，并明确选择了“完整按旧页面还原”，且登录页也要一起恢复。

## 目标

- React 前端恢复为接近旧系统的页面结构、导航组织和视觉层次
- 登录页恢复为旧版后台系统登录体验
- 内页恢复旧版顶部模块导航、左侧模块菜单、页签栏和主内容区关系
- 在不改后端接口协议的前提下，让现有 React 页面呈现旧系统观感

## 非目标

- 不回退 React 路由或重新启用 Flask 模板页面
- 不改动现有接口协议和业务数据结构
- 不新增本次需求之外的新交互或新视觉设计
- 不做无关页面重构

## 现状与可复用来源

仓库里仍然保留了旧版 UI 的关键来源，可以作为 React 还原依据：

- 主框架来源：`templates/base.html`
- 模块导航来源：`templates/partials/app_nav.html`
- 旧页签行为来源：`static/js/app_tabs.js`
- 旧视觉样式来源：`static/css/style.css`
- 旧登录页来源：`templates/login.html`

同时，React 前端当前已经具备可复用的业务能力：

- 登录态与权限：`frontend/src/api/auth.ts`
- 导航数据：`frontend/src/api/query.ts` 中的 `fetchNavigation`
- 页面路由骨架：`frontend/src/router/index.tsx`
- 各业务页数据加载逻辑：`frontend/src/pages/**`

因此本次改造应采用“保留 React 行为，恢复旧版 UI 骨架”的方式，而不是回退技术栈。

## 设计原则

### 1. 结构优先于美化

这次需求的成功标准不是“更好看”，而是“像原来的系统”。因此优先恢复旧版布局关系：

- 顶部模块导航
- 左侧菜单
- 页签栏
- 主内容区
- 登录页表单布局

### 2. 保留 React 路由与数据流

现有路由、鉴权、接口调用逻辑继续保留，只调整组件结构和样式，不把页面重新改回模板渲染。

### 3. 只抽取必要旧样式

旧版 `static/css/style.css` 体量较大，直接整包搬入 React 风险较高。本次只抽取与下列区域直接相关的样式：

- 登录页
- 顶部模块导航
- 左侧菜单
- 页签栏
- 主内容区容器
- 基础按钮、输入框、表格与内容面板

### 4. 业务页做容器级还原

优先还原页面骨架和内容容器，不在本次额外重写每个业务页的内部交互。只要当前页面内容进入旧版容器体系并维持旧风格即可。

### 5. 公共层先统一，再逐页补差异

这次不采用“所有页面全部单独重写”的方式，而是先把 React 前端的公共页面层收口到旧版结构，再只对差异特别大的页面做逐页贴近。这样可以在保持还原度的同时，避免重复堆叠大量近似 DOM 和样式。

## 方案对比

### 方案 A：结构复刻 + 定向样式抽取

做法：

- 按旧版模板结构重建 React 外层框架
- 从旧 `style.css` 抽取必要样式片段
- 现有页面内容挂载到新的旧版容器中

优点：

- 还原度高
- 风险可控
- 不需要重写业务逻辑

缺点：

- 需要将旧模板语义翻译成 React 组件

### 方案 B：整包复用旧 CSS

做法：

- 大量复用旧 CSS
- React 页面尽量套旧类名

优点：

- 初始还原速度快
- 视觉一致性可能更接近旧版

缺点：

- 容易引入历史副作用
- 维护性较差
- 很难保证只影响 React 当前页面

### 方案 C：仅视觉模仿

做法：

- 不复用旧 CSS
- 只参考旧版视觉重新写一套样式

优点：

- 代码更干净

缺点：

- 很可能“不够像原来的”
- 与用户当前目标不一致

## 选型

采用 **方案 A：结构复刻 + 定向样式抽取**。

这是当前最符合用户目标的最小方案：

- 既尽量接近旧系统
- 又不回退前后端分离架构
- 同时避免整包引入旧 CSS 的大范围副作用

## 目标页面结构

### 登录页

登录页恢复为旧后台系统样式，重点包括：

- 更接近旧版的页面居中表单结构
- 标题、表单、按钮和错误提示的旧版视觉层次
- 去掉当前新的营销化说明文案和大面积渐变卡片风格

React 中继续保留：

- 用户名密码受控输入
- 提交态禁用
- 登录失败错误提示
- 登录成功后的原跳转逻辑

### 内页主框架

React `AppShell` 恢复为旧版页面骨架，结构目标如下：

- 顶部：产品模块导航
- 左侧：当前模块菜单
- 中间：页签栏 + 主内容区
- 移动端：底部模块导航

其中：

- 顶部模块导航和左侧菜单的结构参考 `templates/partials/app_nav.html`
- 页签区的布局关系参考 `templates/base.html`
- 移动端导航保留旧模板中的底部模块导航思路

### 页签栏

本次页签栏至少恢复旧版视觉与基础使用体验，包括：

- 页签横向排列
- 当前页签高亮
- 刷新/关闭操作的旧版位置和风格

如果当前 React 架构下完整复刻旧 iframe 多页签成本过高，则允许采用“React 路由页签状态”作为实现方式，只要用户可见体验接近旧版即可。

这里的关键原则是：

- 优先恢复用户感知到的旧版页签栏
- 不为了复刻实现细节而引入不必要复杂度

## 组件调整设计

### `frontend/src/pages/LoginPage.tsx`

调整为旧版登录页 DOM 结构与样式，不改登录提交逻辑。

### `frontend/src/layouts/AppShell.tsx`

从当前的新侧栏卡片布局，改为旧版后台框架：

- 顶部模块导航区
- 左侧菜单区
- 页签栏区
- 主内容承载区

同时保留当前：

- 导航数据拉取
- 退出登录
- 加载态与错误态展示

视觉结构的成功标准是：

- 顶部模块导航、左侧菜单、页签栏和主内容区的关系优先接近旧系统
- 不再保留当前“品牌信息块 + 顶部轻量栏”的新布局气质
- 页签继续使用 React 路由状态实现，但用户感知应接近旧版页签条

### `frontend/src/components/nav/AppMenu.tsx`

从当前按模块分组的卡片式菜单，调整为旧版导航结构：

- 顶部模块导航
- 当前模块的侧栏菜单
- 移动端底部模块导航

需要尽量复用已有导航数据字段，不新增后端接口。

### `frontend/src/components/query/QueryTable.tsx`

统一收口到旧版结果区表格风格，重点恢复：

- 结果标题、统计文本、表格体三层关系
- 更接近旧版的紧凑表头、边框和空数据提示
- 表格作为内容区一部分呈现，而不是独立展示型卡片

### `frontend/src/components/feedback/ErrorState.tsx` 与 `LoadingState.tsx`

调整为旧内容面板里的状态块，而不是当前偏独立卡片式反馈。目标是：

- 加载、报错、空状态都像页面内部状态
- 在查询页和后台页里与内容容器保持一致视觉语言
- 避免出现“一页内多个风格体系并存”

### 各业务页容器

各页面优先做以下统一收口：

- 去掉当前过新的 hero、大卡片、渐变区块
- 调整为旧版页面标题、筛选区、内容面板、结果区结构
- 保留现有数据请求和业务行为

对于首页类页面，允许将当前摘要卡片简化为旧版内容面板式排列，而不是保留当前仪表盘化布局。

## 公共层还原策略

### 页面头部

查询页和后台页的页面头部统一改成旧系统风格：

- 主标题
- 简短页面说明
- 必要时在右侧展示当前账套、状态或已选员工数等辅助信息

目标是保留信息表达，但去掉偏摘要页、概览页的展示感。

### 查询条件区

通用查询页结构优先向旧模板中的筛选容器对齐，采用：

- 字段区
- 勾选项区
- 操作按钮区

三者分层的方式，而不是单纯平铺在同一张新式 surface 上。

### 结果区

查询结果区统一向旧模板中的结果面板靠拢，采用：

- 结果标题
- 结果元信息
- 表格或结果主体

的结构层次。即便内部仍是 React 渲染表格，用户看到的关系也应接近旧版模板。

### 状态反馈

加载、报错、空数据统一变成旧内容区里的提示块，不再保留明显独立卡片感。目标是：

- 页面在加载时像“内容还没出来”，不是切换到另一种界面
- 页面出错时像“当前内容区报错”，不是跳出一张新风格卡片

## 重点页面逐页贴近策略

### `frontend/src/pages/query/QueryHomePage.tsx`

当前页面仍保留较明显的首页 hero 与概览卡片表达，不符合旧系统气质。本次调整为：

- 去掉明显 hero 感
- 管理人员信息进入普通旧版内容面板
- 摘要指标改成更紧凑、更像后台业务统计块的排列

目标是让查询首页更像旧系统里的工作台页面，而不是新前端概览页。

### `frontend/src/pages/query/SummaryDownloadPage.tsx`

该页需要按旧模板的多分区结构单独还原，核心层次为：

- 下载条件区
- 报表说明区
- 指标区
- 自定义表头区
- 下载说明区

保留现有下载逻辑和状态管理，只调整 DOM 结构、内容分区和样式层次。

### `frontend/src/pages/admin/AdminDashboardPage.tsx`

当前后台首页更像迁移阶段占位页，与旧后台入口工作区差距较大。本次调整为：

- 改成更接近旧后台入口页的工作区表达
- 强化入口面板与管理语义
- 不新增新功能，只恢复旧版结构和观感

目标是让后台首页看起来像旧系统管理入口，而不是仅显示两张摘要卡片。

### 其他查询页与后台资源页

其他页面优先依赖公共层自动收口，只在下列情况下做单页最小修正：

- 仍保留明显新的卡片化或 hero 结构
- 筛选区、结果区或说明区与旧模板差异过大
- 页面操作区层次明显不像旧版后台页

不因为“顺手看见”而把所有页面拆开重写。

## 样式组织

建议新增一个 React 侧的旧版 UI 样式文件，例如：

- `frontend/src/styles/legacy-ui.css`

该文件只承载本次恢复需要的旧版样式抽取与适配，不直接复制完整旧 CSS 文件。

样式来源处理原则：

- 原样复用旧类名时，尽量保持命名一致，降低对照成本
- 只抽取当前 React 页面真正使用到的选择器

## 验收标准

- 任一查询页或后台页打开后，整体观感应优先像旧系统，而不是新的 React 展示页
- 页面头部、筛选区、结果区、说明区之间具备清晰旧版层次
- 首页类页面不再保留明显 hero、营销化或展示型概览表达
- 加载、报错、空状态与主内容区风格一致，不出现独立的新式反馈卡片
- 通用组件样式统一，不同页面不会呈现出多个彼此割裂的视觉体系
- 对必须的局部差异做小范围适配，不额外扩散

## 数据流与状态设计

本次不改变 API 调用方式，只调整 UI 使用方式：

- 登录页仍走 `login()`
- 主框架仍通过 `fetchNavigation()` 拉取导航
- 页面内容仍由各自页面自行拉数

页签栏若采用 React 状态方式实现，则状态来源可以是：

- 当前路由
- 最近访问路由列表
- 每个路由对应的标题

但本次不要求把旧 `static/js/app_tabs.js` 全量迁移为复杂的 iframe 管理系统，除非验证后发现不这样做就无法满足旧版体验。

## 错误处理

本次只保留现有必要错误处理：

- 登录失败错误提示
- 导航加载失败提示
- 页面数据加载失败提示

不新增额外复杂兜底逻辑。

## 测试与验证

完成后至少验证以下内容：

- 登录页视觉与旧版风格是否一致
- `/employee/*` 与 `/admin/*` 页面是否统一进入旧版框架
- 顶部模块导航是否正确显示和切换
- 左侧菜单活动态是否正确
- 页签栏是否展示当前页面且支持基础切换
- 现有典型页面能正常加载数据

建议测试范围：

- 现有前端测试中与路由和鉴权相关的部分
- 必要的手工浏览器验证

## 风险与取舍

### 风险 1：旧 CSS 依赖面较广

处理方式：

- 不整包搬运
- 仅抽取命中的样式片段

### 风险 2：旧页签脚本依赖 iframe 模型

处理方式：

- 先恢复视觉和基础交互
- 仅在确有缺口时再补更完整行为

### 风险 3：业务页当前结构与旧模板不完全一致

处理方式：

- 先统一外壳和内容容器
- 只调整与“像旧版”直接相关的页面层级

## 成功标准

当满足以下条件时，本次需求视为完成：

- 用户打开登录页时，视觉和布局明显回到旧系统风格
- 用户登录后看到的是旧版后台式框架，而不是当前新卡片风格
- 顶部模块导航、侧栏菜单、页签栏和主内容区关系恢复
- 主要业务页不再呈现当前“新设计语言”
- React 路由和现有接口仍可正常工作

## 实施范围总结

本次实现聚焦于以下文件和区域：

- `frontend/src/pages/LoginPage.tsx`
- `frontend/src/layouts/AppShell.tsx`
- `frontend/src/components/nav/AppMenu.tsx`
- `frontend/src/pages/query/*`
- `frontend/src/pages/admin/*`
- 新增或调整 React 侧旧版 UI 样式文件

这是一轮 **UI 还原**，不是功能重构。
