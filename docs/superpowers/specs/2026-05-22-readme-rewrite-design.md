# README 重写设计

## 目标

把当前 `README.md` 重写成一份同时服务于开发者和部署者的综合文档：

- 新同事第一次打开仓库时，能在前半部分快速完成本地启动。
- Windows 部署和运维人员能在后半部分直接找到安装、升级、回滚步骤。
- 现有命令、初始化方式、旧库升级方式与当前代码保持一致。

## 受众

### 1. 开发者

需要快速找到：

- 本地环境要求
- 安装依赖
- 初始化数据库
- 初始化管理员
- 本地启动方式
- 常用测试命令

### 2. 部署者 / 运维人员

需要快速找到：

- Windows 部署准备
- 首次部署
- 旧数据库升级
- 生产启动方式
- 服务安装
- 备份与回滚

## 采用方案

采用“双入口 README”：

- 前半部分为开发者快速上手
- 后半部分为 Windows 部署与运维手册

这样不拆文档，也不把部署细节塞进开头，阅读路径更清楚。

## 文档结构

新版 README 使用以下结构：

1. 项目简介
2. 功能概览
3. 快速开始
4. 常用命令
5. 项目结构
6. Excel 导入说明
7. Windows 部署与运维
8. 开发与测试
9. 补充说明

## 文案原则

- 以“直接可执行”为主，不写营销式描述。
- 每个章节优先回答“什么时候看这一节”。
- 命令块尽量完整可复制。
- 对高风险误用点直接写清楚：
  - `python app.py` 仅用于本地开发
  - 生产环境不要依赖启动时自动升级 schema
  - 旧库升级使用 `flask --app manage.py upgrade-legacy-schema`

## 范围边界

这次只重写 `README.md` 的内容与结构，不改：

- 代码
- 部署脚本
- `docs/` 下其他现有说明文件

## 一致性要求

README 中的说明必须与当前项目实现保持一致：

- 应用启动不会隐式执行 schema 升级
- 新环境初始化使用 `init-db`
- 旧库兼容升级使用 `upgrade-legacy-schema`
- 生产启动使用 `waitress`

## 非目标

- 不把 README 拆成多个独立文档
- 不在 README 中展开业务规则或数据库设计细节
- 不把 AI/插件集成说明放到主阅读路径前面
