# 账套厂休日期配置与假期重叠扣减设计

## 背景

当前账套只有 `factory_rest_days` 总天数字段，能够参与管理人员考勤中的总量计算，但无法表达“这个月具体哪几天是厂休”。

这会带来两个问题：

1. 无法在账套界面直接维护具体厂休日期，只能手工填写总数。
2. `出差 / 婚假 / 丧假` 这类会加到出勤天数的假期，当前按请假记录自然日重叠时长直接折算天数，若其中覆盖了厂休日，会与厂休重复计算。

本次设计目标是在不推翻现有总数计算链路的前提下，为账套补充“厂休日期明细”，并让这些日期参与上述假期的重叠扣减。

## 目标

1. 管理员可以在账套界面为某个月选择具体厂休日期。
2. 系统根据厂休日期明细自动汇总 `factory_rest_days` 总数。
3. 现有所有仅依赖 `factory_rest_days` 总数的逻辑保持兼容。
4. `出差 / 婚假 / 丧假` 在计入出勤天数前，需扣除与厂休日期重叠的部分，避免同一天重复计入。
5. 重叠扣减支持半天粒度。

## 非目标

1. 本次不改变事假、病假、调休、工伤的统计口径。
2. 本次不改动其他月份共享厂休日历；厂休日期只归属于单个账套月份。
3. 本次不设计跨月厂休模板、批量复制厂休规则等扩展能力。
4. 本次不直接修改业务代码，仅确认设计与规则边界。

## 设计方案

### 1. 数据模型

采用“日期明细为主、总数为辅”的方案。

- 账套继续保留 `factory_rest_days` 字段，作为兼容现有计算链路的总数缓存。
- 新增厂休日期明细存储，归属于单个账套。
- 每条明细至少包含：
  - `account_set_id`
  - `rest_date`
  - `rest_unit`

其中 `rest_unit` 表示当天厂休粒度，约束为：

- `1.0`：全天厂休
- `0.5`：半天厂休

为避免歧义，半天厂休需要区分时段，建议额外记录：

- `rest_period = "am" | "pm" | "full"`

对应关系如下：

- `full` 对应 `rest_unit = 1.0`
- `am` / `pm` 对应 `rest_unit = 0.5`

这样做的原因是：用户已经明确“出差算半天，厂休半天”，仅有 `0.5` 数值还不足以判断重叠发生在上午还是下午，无法与半天假期精确比对。

### 2. 账套界面

账套中心保留原有“厂休天数”展示，但交互改为“由日期明细自动汇总”。

建议界面行为：

1. 选择账套后，展示该月日历或日期列表。
2. 管理员可将某天标记为：
   - 非厂休
   - 上午厂休
   - 下午厂休
   - 全天厂休
3. 页面实时汇总并展示 `factory_rest_days`。
4. 原有数值输入框改为只读展示，或改成“自动汇总：X 天”，避免明细与总数双向编辑产生冲突。

### 3. 账套保存规则

- 创建账套时，允许不配置厂休日期，默认总数为 `0`。
- 编辑账套时，以提交的厂休日期明细为准，服务端重新计算并覆盖 `factory_rest_days`。
- 锁定账套后，厂休日期明细与汇总总数都不可再修改，沿用现有封账约束。

服务端汇总规则：

- 全天厂休累计 `1`
- 上午厂休累计 `0.5`
- 下午厂休累计 `0.5`

最终汇总写回账套 `factory_rest_days`，供现有管理人员考勤、查询页、首页汇总等继续使用。

### 4. 假期重叠扣减规则

仅以下假期类型参与“与厂休重叠扣减”：

- 出差
- 婚假
- 丧假

原因：当前系统中这三类假期会直接加到出勤天数，若厂休与它们重叠，会造成重复记入。

处理顺序应为：

1. 先计算假期在当月内的原始有效天数。
2. 再计算该假期与账套厂休日期的重叠天数。
3. 最终用于 `attendance_days` 累加的值 = 原始有效天数 - 重叠天数。

扣减后结果不得小于 `0`。

### 5. 半天重叠口径

重叠扣减按半天粒度计算，必须识别上午 / 下午。

规则如下：

- 假期覆盖全天，且厂休为全天：扣 `1`
- 假期覆盖上午，且厂休为上午：扣 `0.5`
- 假期覆盖下午，且厂休为下午：扣 `0.5`
- 假期覆盖全天，厂休仅半天：扣 `0.5`
- 假期仅半天，厂休为全天：扣 `0.5`
- 假期半天与厂休半天时段不同：扣 `0`

如果现有请假记录只有起止时间，没有显式“上午/下午”标签，则实现时通过时间区间映射到上午/下午/全天口径。

### 6. 与现有总量计算的关系

本次设计不推翻现有 `factory_rest_days` 的总量用途，而是在其前面补充一层明细来源。

保留不变的逻辑：

1. 所有基于 `factory_rest_days` 总数做差额计算的流程继续按总数运行。
2. 管理人员考勤 `absence_gap = 月天数 - attendance_days - factory_rest_days` 的总量公式保持不变。
3. 首页、查询页、导出逻辑中对厂休总天数的读取方式保持兼容。

新增变化仅在于：

1. `factory_rest_days` 不再建议手输，而由明细自动汇总。
2. `business_trip_days / marriage_days / funeral_days` 在参与 `attendance_days` 前，需要先扣除与厂休明细的重叠部分。

### 7. 练义炜案例口径

以练义炜 `2026-04` 的出差记录为例：

- 出差记录：`2026-04-08 08:00:00` 至 `2026-05-01 17:00:00`
- 当前系统会按当月自然日重叠折算出差天数

引入厂休日期明细后：

- 若账套中将该区间内部分日期标记为厂休，则这些厂休日与出差重叠的部分，需要从原始出差天数中扣除。
- 扣减后的出差天数，才参与 `attendance_days = actual_attendance_days + business_trip_days + marriage_days + funeral_days`

也就是说，厂休不仅影响“月天数差额计算”，也影响“可加回出勤天数的假期天数”本身。

## 风险与约束

1. 现有假期记录主要是起止时间，半天重叠的准确判断依赖统一的上午/下午划分规则；实现时必须先明确这一规则。
2. 若允许同时手工编辑总数和日期明细，极易出现数据冲突；因此应避免双入口编辑。
3. 厂休日期属于账套参数，需遵守账套锁定规则，防止结算后口径漂移。
4. 如果已有历史账套只填写了总数，没有日期明细，后续实现时需要兼容“旧数据仅有总数”的只读场景。

## 验证方式

1. 在账套界面为某个月设置若干全天和半天厂休后，页面显示的厂休总数与明细累计一致。
2. 保存后重新进入账套，厂休日期明细和总数都能正确回显。
3. 锁定账套后，厂休日期明细不可编辑。
4. 对存在 `出差 / 婚假 / 丧假` 的员工，当假期覆盖厂休日期时，对应天数会按半天 / 全天规则扣减。
5. 对不覆盖厂休日期的假期，统计结果保持不变。
6. 仅依赖 `factory_rest_days` 总数的现有页面与计算，不因引入明细而失效。
